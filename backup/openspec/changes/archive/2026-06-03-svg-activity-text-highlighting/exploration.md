## Exploration: svg-activity-text-highlighting

### Current State
Iteration 1 already established the semantic boundary: runtime emits `last_event`, ordered `executed_activities`, and `transition_path_ids`, while the viewer still highlights only `active_ids` plus taken external transition edges. Render currently outputs visible lifecycle/internal/transition text through Graphviz HTML-like labels, but it does not expose any render-owned contract for targeting individual text fragments, and public `id=` usage is intentionally reserved for authored highlightable states/transitions only.

### Affected Areas
- `src/mbse/application/prepare_hsm_render.py` — render prep already knows the authored lifecycle/internal/transition structure and is the right place to derive fragment-target metadata.
- `src/mbse/infrastructure/templates/hsm_v1.dot.j2` — Graphviz label markup must emit private fragment markers for label parts without adding new public authored IDs.
- `src/mbse/application/build_session.py` — likely needs a render step that converts SVG marker hints into viewer-consumable fragment targets.
- `src/mbse/infrastructure/svg_contract.py` — likely needs SVG parsing helpers beyond top-level `id` extraction if marker hints are normalized after render.
- `src/mbse/domain/models.py` — app/session DTOs need a render-owned text-target contract separate from runtime `last_event` semantics.
- `src/mbse/application/web_runtime_bridge.py` — should pass render-owned target metadata through unchanged while keeping runtime semantics separate.
- `src/mbse/web/static/viewer.js` and `src/mbse/web/static/viewer.css` — viewer must resolve semantic refs plus render targets into precise, legible text highlighting.
- `tests/support/viewer_browser_harness.py` and `tests/integration/test_mbse_workbench.py` — browser-facing tests must move beyond whole-group ID highlights and assert fragment-level text targeting.

### Approaches
1. **Graphviz marker attributes only** — embed private markers directly in HTML-like labels and let the viewer query raw SVG anchors/attributes.
   - Pros: Smallest pipeline change; no SVG rewrite step.
   - Cons: Graphviz does not preserve nested HTML-label `ID` attributes as stable SVG IDs, raw `HREF` markers introduce link semantics/namespace-selector fragility, and viewer/tests become coupled to Graphviz-emitted wrapper details.
   - Effort: Medium

2. **Render-owned markers plus lightweight SVG normalization** — embed private marker hints in Graphviz labels, then post-process rendered SVG into a stable private fragment-target map for the viewer.
   - Pros: Keeps public authored IDs clean, gives the viewer a stable contract, and preserves the boundary that render owns fragment targeting while runtime stays semantic-only.
   - Cons: Slightly larger slice across render/session/viewer/test plumbing.
   - Effort: Medium

### Recommendation
Use **Approach 2**.

Minimal coherent boundary for iteration 2:
- keep existing active-state highlighting unchanged
- keep external arrow highlighting driven only by `last_event.transition_path_ids`
- add one render-owned text-target contract for lifecycle labels/activities, transition event+activity text, and internal-transition row text
- let the viewer combine that contract with `last_event` semantics; do not change runtime payload meaning and do not move viewer concerns into authored HSM IDs

Recommended render-owned contract:
- top-level session/app metadata, not runtime `last_event`
- keyed by authored semantic refs, for example:
  - lifecycle slot: `{owner_kind, owner_id}` → section-label target + activity targets by `activity_id`
  - external transition: `{transition_id}` → event-text target + activity targets by `activity_id`
  - internal transition: `{internal_transition_id}` → `internal_transitions:` section target + row event/activity targets
- values should be private fragment refs/selectors generated by render, not public authored SVG IDs

Post-processing answer:
- Graphviz label generation alone is **not** a safe final target contract for this slice
- it can carry private marker hints, but Graphviz ignores nested HTML-label `ID` attributes and auto-generates wrapper groups
- a lightweight normalization step after SVG render is the safest way to turn those hints into stable non-public fragment targets

Viewer highlight strategy:
- keep current state/edge classes for shape highlights
- add separate text-fragment highlight classes with color + underline/background accent
- avoid heavier font weight
- prefer a thin underline plus subtle warm background for active text fragments so readability stays high on dense labels

Review-size forecast:
- likely **320-430 changed lines** across models/render/template/session/viewer/harness/tests
- this is near the 400-line review budget, so proposal/tasks should treat size risk as **Medium-High** and be ready to split if SVG normalization plus browser coverage grows

### Risks
- **Graphviz marker behavior**: private marker hints must be validated against real Graphviz output for node labels, edge labels, and compound-state labels.
- **Boundary drift**: it is easy to leak render targets into runtime snapshots or authored public IDs unless the contract stays top-level and render-owned.
- **Browser-test cost**: the current harness mostly understands ID/class targeting; fragment selectors or normalized private targets will require harness expansion.
- **Legibility regression**: broad text emphasis or font-weight changes will hurt dense state labels quickly.
- **Review budget**: if the slice also redesigns viewer styling or over-generalizes fragment APIs, it will likely exceed the 400-line budget.

### Ready for Proposal
Yes — propose a change that adds a render-owned private fragment-target contract plus viewer-side text highlighting only, explicitly excludes runtime semantic changes and authored public-ID expansion, and flags a likely **Medium-High** size risk near the 400-line review budget.
